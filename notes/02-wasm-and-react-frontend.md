# 02 - WASM Build, React Frontend & Web Audio API

## What wasm-pack produces

Running `wasm-pack build --target web --out-dir pkg` inside `engine/` compiles the Rust
library to WebAssembly and generates a ready-to-use JS package in `engine/pkg/`:

| File | Purpose |
|---|---|
| `engine_bg.wasm` | The compiled Rust code as a WebAssembly binary |
| `engine.js` | Auto-generated JS glue - loads the WASM, handles memory marshalling |
| `engine.d.ts` | TypeScript types for our exported functions |
| `package.json` | Makes `pkg/` importable as a local npm package |

The `pkg/` folder is a build artifact - it goes in `.gitignore` and is regenerated by running
`wasm-pack build` again.

### Linking it to the frontend

```bash
cd web
npm install ../engine/pkg
```

This adds `"engine": "file:../engine/pkg"` to `package.json` and makes it importable like any npm package.

## Vite configuration for WASM

Two things are needed in `vite.config.ts`:

**1. `vite-plugin-wasm`**
The WASM glue code uses `new URL('engine_bg.wasm', import.meta.url)` to locate the binary at
runtime. Without this plugin, Vite doesn't handle that URL pattern correctly and the browser
gets a 403 or compile error when trying to fetch the `.wasm` file.

**2. `optimizeDeps.exclude: ['engine']`**
Vite pre-bundles dependencies with esbuild for faster startup. This breaks the WASM package
because esbuild doesn't understand the runtime URL pattern. Excluding it tells Vite to leave
the package alone and let the browser handle it natively.

**3. `server.fs.allow: ['..']`**
Vite's dev server blocks requests to files outside the project root (`web/`) by default.
Since `engine/pkg/` sits one level up, we have to explicitly allow it.

## Web Audio API - how the microphone pipeline works

The Web Audio API models audio processing as a **graph of nodes**, all living inside an
`AudioContext`. Audio flows from sources through processing nodes to a destination.

Our graph:

```
Microphone (getUserMedia stream)
    -> MediaStreamSourceNode   wraps the mic stream as a graph node
    -> ScriptProcessorNode     fires a callback with raw PCM samples
    -> AudioContext.destination  speakers (connected to keep the graph alive, but no audible output)
```

### Key concepts

**AudioContext**
The root object. Everything else is created from it. It also tells us the sample rate
the hardware is running at (usually 44100 Hz, sometimes 48000 Hz).

**getUserMedia**
A browser API that requests microphone access. The browser shows a permission prompt.
Returns a `MediaStream` if the user allows it.

**MediaStreamSourceNode**
Wraps a `MediaStream` (mic input) so it can be connected into the Web Audio graph.

**ScriptProcessorNode**
Deprecated but simple. Fires `onaudioprocess` every time it has collected `BUFFER_SIZE`
new samples. The event gives us a `Float32Array` of PCM data in [-1.0, 1.0] - exactly
what our Rust function expects.

We use a buffer size of 4096 samples. At 44100 Hz that's ~93ms per frame.

**Why connect to `ctx.destination`?**
The ScriptProcessorNode only fires if it's connected downstream in the graph. We connect
it to the speakers but produce no output, just to keep the graph active.

### The TODO: migrating to AudioWorklet

ScriptProcessorNode runs on the **main thread** alongside React, which means heavy audio
processing could cause UI jank, and the browser can throttle it when the tab is backgrounded.

`AudioWorklet` runs on a dedicated audio thread - lower latency, more reliable. The migration
when we're ready:
1. Write a worklet processor file that imports and runs the WASM detection
2. Use a `SharedArrayBuffer` ring buffer to pass detected notes back to the main thread
3. Replace `createScriptProcessor` with `new AudioWorkletNode(ctx, 'pitch-detector')`

## How the React component is structured

`App.tsx` uses two pieces of React state and two refs:

**State** (causes re-renders when changed):
- `status` - `'idle' | 'loading' | 'listening' | 'error'`
- `note` - the current detected note string, e.g. `"E2"`

**Refs** (hold values across renders without triggering re-renders):
- `audioCtxRef` - the AudioContext, needed to close it on stop
- `processorRef` - the ScriptProcessorNode, needed to disconnect it on stop

We use refs for the audio objects because we never want a state change to cause React to
recreate them - that would interrupt the audio stream.

**`useCallback`**
`startListening` and `stopListening` are wrapped in `useCallback` so React doesn't recreate
them as new function objects on every render. Not critical here but good practice.

## Initialising WASM

```ts
import init, { detect_note } from 'engine'

await init()  // fetches and compiles engine_bg.wasm
detect_note(samples, sampleRate)  // now safe to call
```

`init()` must complete before any exported Rust functions are called. We do it once at
the start of `startListening`. Subsequent calls to `detect_note` are synchronous and fast.
